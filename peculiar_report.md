## Overview

Peculiar's submission has the strongest backend architecture in the cohort and the most carefully structured frontend codebase — but the two are not connected. The README is honest about this: *"The frontend currently still uses local UI snapshot data in `frontend/src/lib` for most screens. Full frontend-to-backend runtime integration can be completed next without changing the overall screen structure."* Every screen renders with interactive local state (tickets can be "created", comments can be "added", customers can be "added") but no API call is ever made. The backend is production-quality code that runs in isolation. The disconnection is the central issue.

### Architectural Notes on the Backend

**Service layer (`services.py`):** Business logic — ID generation, dashboard snapshot building, workspace snapshot building, pagination, seed data — is completely separated from views. Views are thin: receive request, call service, serialize, return. This is the cleanest backend architecture in the cohort.

**`public_id` system:** Tickets use human-readable IDs (`T-001`) and reference codes (`#1001`) generated by scanning the max existing ID and incrementing. This mirrors how real CRM tools expose ticket references.

**`TicketWorkflowState` vs `TicketStatus`:** A separate workflow concept (`to_do`, `in_progress`, `overdue`, `done`) is tracked independently from status (`open`, `in_progress`, `resolved`, `closed`). This is the only submission that models this distinction.

**CORS:** `CORS_ALLOWED_ORIGINS` with specific origins (`127.0.0.1:3000`, `localhost:3000`) — the only submission to use specific allowed origins rather than `CORS_ALLOW_ALL_ORIGINS = True`.

**DB efficiency:** `Ticket.objects.select_related("customer").prefetch_related(Prefetch("comments", ...))` — avoids N+1 queries on the list view. No other submission does this.

**Serialization:** All JSON keys are camelCase (`workflowState`, `createdAt`, `referenceCode`, `authorName`) to match the frontend TypeScript types directly. No snake_case/camelCase mismatch.

### Frontend Architecture

The frontend is the most well-structured in the cohort. Notable patterns:

**Route group:** Pages are organized under `app/(workspace)/` — a Next.js App Router route group that applies a shared layout without affecting the URL path. This is proper App Router usage.

**Component decomposition (~30 components):** Each feature area has dedicated components: `TicketsWorkspace`, `TicketsList`, `TicketListItem`, `TicketDetailsWorkspace`, `TicketDetailsCard`, `CommentThread`, `AddCommentForm`, `NewTicketModal`, `CustomersWorkspace`, `CustomersTable`, `CustomerRow`, `NewCustomerModal`, etc. No feature logic bleeds across areas.

**`useDeferredValue` for search:** Customer search uses React 19's `useDeferredValue` to defer the filter computation — avoids blocking the input on large lists.

**Functional pagination:** Customers page has real working pagination with page size selector, page navigation, and select-all/individual row checkboxes — all wired through local state correctly.

**Interface contracts match API:** The TypeScript types in `lib/types.ts` match the backend serializer output exactly: `workflowState`, `referenceCode`, `authorName`, `createdAt` — camelCase throughout. The API was clearly used as the design source for the component interfaces.

### The Disconnect

`lib/data.ts` is the only place where this submission falls. Instead of:
```ts
const response = await fetch('/api/tickets/workspace/');
```

every page calls:
```ts
const snapshot = getTicketWorkspaceSnapshot(); // returns seededTickets
```

The `NewTicketModal.onCreate` callback receives a `NewTicketInput` and constructs a fake ticket object locally. The `handleAddComment` in `TicketDetailsWorkspace` appends to `ticket.comments` in state. The component interface is designed to accept data from the API — the wiring simply was not completed.

### Design Fidelity

Confirmed strong by tester — the UI closely matches the Figma reference including the real NativeTalk logo (`frontend/public/nativetalk-logo.png`). The colour `#3bc508` is used throughout (vs Figma's `#3EBF0F` — close but not exact). The layout (fixed sidebar, workspace content area, cards) mirrors the Figma structure. Components are cleanly styled with Tailwind and CSS custom properties (`var(--border)`, `var(--surface)`, `var(--muted)`). The real company logo and overall visual language place this submission among the strongest for design fidelity in the cohort.

### 1. Speed vs. Accuracy — 13 / 25

Did the candidate deliver a functional MVP within the time limit without sacrificing structural integrity?

**Assessment:**
The backend is fully functional — all endpoints work, data persists, seed data loads. The frontend renders all four required screens and is interactive. But no API call is ever made from the frontend — data is mock, mutations are in-memory. A working full-stack MVP was not delivered within the time limit.

Credit is given for: all screens rendering correctly, correct framework (Next.js 16), and confirmed strong design fidelity including the real NativeTalk logo. The gap is fundamental — no user action has any backend effect, nothing persists on refresh, and the frontend/backend might as well be two separate submissions.

### 2. Prompting & Context Management — 19 / 25

Did the candidate pass the Django schema to the Next.js agent? Did they feed the Figma design system to the AI?

The main gap: the API integration step was never completed. The prompt chain clearly ended before wiring `lib/data.ts` to real fetch calls.

**Score: 19 / 25 (76%)**

## Summary

Peculiar built the strongest backend in the cohort: service layer separation, camelCase serialization, `public_id` system, scoped CORS, N+1-safe queries, a `seed_demo_data` command, and a dedicated workspace endpoint. The frontend is the most architecturally sound: ~30 components organized by feature, proper Next.js route groups, functional pagination, and TypeScript types that exactly mirror the backend contract.

The critical gap is that the frontend was never wired to the backend. Every screen shows mock data. Every user action mutates only local React state. The README acknowledges this openly. The component interfaces are ready for integration — `lib/data.ts` returning mock snapshots is the only thing standing between this submission and a fully functional product.

**Strongest area:** Architectural Oversight — best backend architecture and CORS handling in the cohort
**Weakest area:** Speed vs. Accuracy — polished backend and frontend that never talk to each other
